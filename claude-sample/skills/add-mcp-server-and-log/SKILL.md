# MCPサーバー導入＆ログ記録スキル

ユーザーから「MCPサーバーを導入したい／検討したい」という相談を受けた際に、
要否の検討からコマンド実行・反映確認・ログ記録までを一連の流れとして行うスキル。

## 実施手順

### 1. 導入要否を検討し、ユーザーに提案する

- ユーザーの今後の作業内容（例: フォーム入力など対話的な操作が増える見込みか）を確認する。
- MCPサーバーを使う方法と使わない方法（スクリプトを書いて実行する等）を比較し、
  それぞれのメリット・デメリットを整理してユーザーに提示する。
  - MCPなし方式のデメリット例：対話的な調整がしにくい、試行錯誤コストが高い、
    初回セットアップが必要、失敗原因の把握がしづらい、状態依存の分岐操作に弱い
- ユーザーの作業内容に応じてどちらが向いているか所見を伝え、判断はユーザーに委ねる。

### 2. 現状のMCP構成を確認する

```
claude mcp list
```

併せて、追加対象のツール（例: Playwright）がすでに環境にあるかどうかも調べておく
（例: `npx <package> --version`、`python -c "import <module>"` 等）。

### 3. スコープをユーザーと合意する

`claude mcp add --help` でスコープ仕様を確認し、以下を表などで分かりやすく説明する。

| スコープ | 影響範囲 | 保存場所 |
|---|---|---|
| local（デフォルト） | 現在のプロジェクトディレクトリ限定。他ディレクトリ・他人に影響しない | ユーザー側設定（プロジェクトパス専用として記録） |
| project | プロジェクト全体・チーム共有 | プロジェクト直下の `.mcp.json`（gitにコミットされる） |
| user | 全プロジェクトに影響（グローバル） | ユーザー設定（`~/.claude` 配下） |

- 「他のディレクトリに影響しないか」といった懸念には正確に答える（local は影響しない、project は git管理下で共有される等）。
- ユーザーがスコープを選択するまでコマンドを実行しない。

### 4. ログファイルを作成する（コマンド実行前に経緯を記録）

`logs/<YYYY-MM-DD>_<MCPサーバー名>MCPサーバー導入ログ.md` を作成し、
ここまでの検討経緯（背景、比較した選択肢、環境調査結果、スコープの説明、ユーザーの決定）を記録する。
ユーザーから「やったことは全部ログに残すように」という方針が示されているため、
**コマンド実行前の検討段階から記録する**こと。

### 5. `claude mcp add` を実行する

合意したスコープでコマンドを実行する。スコープ未指定はlocalになる。

```
claude mcp add <name> <command> -- <args...>
# 例: claude mcp add playwright npx @playwright/mcp@latest
```

### 6. 反映結果を直接確認する（claude mcp list は当てにしない）

`claude mcp add` 実行直後は、現在のセッションが起動時の設定をキャッシュしているため
`claude mcp list` / `claude mcp get <name>` が「未設定」と表示されることがある。
その場合は設定ファイルを直接読んで反映を確認する。

```python
import json
with open(r"C:\Users\<user>\.claude.json", "r", encoding="utf-8") as f:
    data = json.load(f)
for k, v in data.get("projects", {}).items():
    mcp = v.get("mcpServers", {})
    if mcp:
        print(k, mcp)
```

- プロジェクトキーは `\` 区切りではなく `/` 区切りで保存されている場合があるので、
  両方の表記で探す、または全件走査して確認する。
- 反映が確認できたら「設定は書き込まれているが、ツールの利用には新しいセッションの開始が必要」
  という点をユーザーに正確に伝える（自分でセッションを再起動することはできない）。

### 7. ログに結果を追記する

コマンド出力、検証方法、確認できた登録内容、`claude mcp list` で表示されなかった理由の考察、
今後のステータス（次回セッションで動作確認が必要、等）を追記する。

### 8. ユーザーへ報告する

「設定の追加は完了し直接確認済みだが、実際に使えるかの動作確認は新しいセッションでないとできない」
という事実を、できたこと／できていないことを分けて正直に伝える。
