# 画面テスト実施＆ログ記録スキル

[[read-test-spec-and-log]] で仕様を理解した後、Playwright MCPを使って実際に1件のテスト項目を実施し、
結果を整理してログに記録するスキル。CLAUDE.mdの「画面テストのルール」「期待値を推測してはいけない」
方針に従う。

## 前提
- Playwright MCPサーバーが利用可能であること（[[add-mcp-server-and-log]]参照）
- 対象のテスト項目（No、検証内容）が仕様書から把握できていること

## 実施手順

### 1. 実施前にログファイルへ手順を記録する

`logs/<YYYY-MM-DD>_結合テストNo<N>実施ログ.md` を作成し、コマンド実行前に以下を記載する。

- 対象テスト項目（仕様書から転記：No、分類、テスト項目、検証内容）
- 対象サイトURL
- 前提確認（MCPサーバーが利用可能であることの確認結果）
- これから実施する手順（番号付きで具体的に）

ユーザーから「やることは全部事前にログに残すように」という方針が示されているため、
**ツールを実行する前に手順をログ化する**こと。

### 2. 各操作の前に「何をしようとしているか」を一言説明できるようにしておく

ユーザーから「これは何をしようとしているの？」と問われることがあるため、
各 `mcp__playwright__browser_*` 呼び出しが「ログのどの手順に対応し、何を確認するためのものか」を
即答できるようにしておく。不明瞭なまま機械的にツールを連打しない。

### 3. ブラウザを操作してテストを実施する

検証内容に応じて、以下のようなPlaywright MCPツールを組み合わせる。

- `browser_navigate`：対象URLへ遷移
- `browser_take_screenshot`（`fullPage: true` 推奨）：CLAUDE.mdの「テスト開始時／主要画面遷移後／失敗時／エラー時」のルールに従い取得
  - 出力先ディレクトリが存在しないと `ENOENT` で失敗するため、事前に存在するパスか `filename` を相対パスのみにして既定の出力先に保存する
- `browser_snapshot`：アクセシビリティスナップショットで実際の表示内容（見出し・フォーム・ボタン・結果欄など）を構造的に確認する（スクリーンショットだけでは「正常に表示されているか」の判定根拠として弱い）
- `browser_console_messages`（`level: "error"`）：コンソールエラー（404、JSエラー等）を確認する
- 必要に応じて `browser_click` / `browser_type` / `browser_fill_form` 等で入力操作・ボタン押下・画面遷移確認を行う
- 最後に `browser_close` でブラウザを閉じる

### 4. 取得した事実を整理し、検証内容と照らし合わせる

- 仕様書の「検証内容」に書かれている文言を基準に、実際の結果が一致するかを確認する。
- **仕様書に書かれていない事象（例：コンソールの404エラー、見た目の細部など）を見つけた場合、
  それを「失敗」や「問題」と勝手に判定しない。** 事実として記録した上で、
  「テスト項目の検証内容には含まれない事象だが、成功として扱ってよいか」をユーザーに確認する。

### 5. ログに結果を追記する

以下を表形式で整理して追記する。

- 実施日
- 検証内容（仕様書記載の文言）
- 実際の結果（観測した事実）
- 期待値との差異
- 発生したエラー（あれば、影響範囲も含めて）
- スクリーンショットの保存パス

最後に「ステータス」として、判定（成功／失敗）が確定したか、ユーザー判断待ちの点があるかを明記する。

### 6. ユーザーへ報告する

CLAUDE.mdの「結果を報告する」項目（実施したテスト項目／成功・失敗／実際の結果／期待値との差異／発生したエラー）
に沿って簡潔にまとめ、判断が必要な点（仕様書に明記されていない事象の扱い等）を明確に分けて確認する。
あわせて、仕様書（xlsxの「実施日」「結果」列等）への結果反映が必要かも確認する。
